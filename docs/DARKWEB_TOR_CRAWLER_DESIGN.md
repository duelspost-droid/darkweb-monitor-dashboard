# JB금융그룹 자사 다크웹 노출 모니터링 — 종합 설계·거버넌스 문서

> 상태: **설계/아키텍처 문서 (v1.0)**. 실제 빌드·운영은 본 문서의 법무·CISO·DPO 사인오프 및 각 단계 진입 게이트 충족 후 별도 진행한다. 본 문서는 리스크 비평에서 지적된 구멍·필수변경·금지사항을 모두 반영한 통합본이다.

## 0. Executive Summary 와 핵심 원칙

### 0.1 요약

JB금융그룹 4개 도메인(`jbfg.com`=JB금융지주, `jbbank.co.kr`=전북은행, `kjbank.com`=광주은행, `wooricap.com`=JB우리캐피탈)의 다크웹/유출사이트 노출을 **수동·읽기전용**으로 조기 탐지하는 **방어적 위협인텔리전스** 역량을 설계한다. 결과만 기존 Supabase(`breach_findings`/`scan_runs`) + Next.js 대시보드로 적재·표시한다.

핵심 설계 결정 4가지:
1. **PII 최소화는 "저장 후 정리"가 아니라 "수집 단계 게이트"다.** 자사 4개 도메인에 매칭되지 않은 바이트는 영속 저장소에 도달하지 못하도록 파이프라인 순서를 코드로 강제한다(스트리밍→매칭→컨텍스트 윈도우→레닥션→정규화→leak-guard→적재).
2. **Tor 직접수집의 ROI는 단 한 점 — "랜섬웨어 DLS 조기경보"에 집중된다.** 인포스틸러·과거 유출은 합법 API가 더 안전·정확하므로 중복 투자하지 않는다. 폐쇄 마켓·인증 포럼은 **합법적으로 자체수집 불가**하므로 상용/금융보안원·금융ISAC 피드로 보완한다(빌드 vs 바이).
3. **단계적 게이트.** Phase 0(클리어넷 공개소스)은 거버넌스·접근통제 보강 후 운영 가능. Phase 1(Tor `.onion`)은 **형사 적법성 법무 의견서·평문/링크 노출 차단 코드·stale-delete tier 격리·보존 컬럼/키셋 정합화 4종을 모두 충족하기 전 단일 파일럿도 금지.**
4. **"대시보드 무변경"은 거짓이었다 — 철회한다.** 코드 검증 결과 대시보드는 평문 `account`를 마스킹보다 우선 렌더하고 `reference_url`을 무조건 클릭 가능 링크로 렌더한다. 이는 PII 노출·유출사이트 직결 클릭 경로이며, **프런트 코드 변경은 Phase 1 필수 산출물**이다.

### 0.2 핵심 원칙 (전 설계 관통)

- **수동·읽기전용 수집**: 이미 공개 게시된(공개 인덱스/유출사이트 리스팅) 정보만 GET. 인증·가입·CAPTCHA·로그인·페이월 게이트 뒤는 접근하지 않으며 우회를 설계·구현하지 않는다.
- **자사 한정 키워드 매칭**: 4개 도메인/사명/브랜드 매칭 항목만 추출·정규화·저장. **전체 덤프 다운로드·보관 금지.**
- **타인 PII 최소화**: 매칭 컨텍스트 외 제3자 PII는 수집 즉시 폐기/레닥션. 평문 비밀번호·민감 식별값 원문 미저장(분류 라벨만).
- **절대 금지(7장 명문화)**: 마켓 거래·덤프 구매, 인증/CAPTCHA 우회, 게시·업로드, 표적 공격, 공격적 탐지회피, 자격증명 유효성 검증.

## 1. 목표 · 범위 · 비범위

### 1.1 목표
- 자사 4개 도메인·임직원·고객 계정의 다크웹/유출 노출을 **미러·피드 반영 전에 조기 탐지**해 계정 탈취·2차 피해(보이스피싱·무단 인출)를 예방한다.
- 노출 사실·규모·날짜·데이터 클래스 분류를 마스킹·라벨 상태로 기존 대시보드에 통합해 보안팀 대응(세션 무효화·비번 리셋·통지 검토)을 지원한다.

### 1.2 범위 (IN)
- 공개 인덱스/리스팅에 **이미 게시된** 메타데이터의 읽기전용 수집.
- 자사 4개 도메인/사명/브랜드 키워드 매칭 항목만.
- 산출: `mkRawFinding` 고정 컬럼 셋 호환 정규화 레코드(메타데이터 + 마스킹 + 분류 라벨 + 출처 포인터).

### 1.3 비범위 (OUT — 설계하지 않음)
- 로그인·가입·CAPTCHA·페이월·초대코드·세션쿠키 등 인증/안티봇 게이트 **뒤 영역 및 그 우회**.
- 범죄 마켓 거래·덤프 구매·샘플 요청, 전체 덤프 다운로드·보관.
- 게시·업로드·상호작용, 표적 공격·침입, 공격적 탐지회피, 타인 계정 로그인·자격증명 유효성 검증.
- 4개 도메인 화이트리스트 밖 키워드(경쟁사·개인 타겟 등).
- 폐쇄 마켓·인증 포럼 영역 자체수집(→ 상용/기관 피드로 분리, 9장).

## 2. 법무 · 거버넌스

> ⚠️ **본 장의 모든 조문 번호·요건은 법무가 축조 확인 후 확정하고 거버넌스 문서 버전에 서명 보존한다.** 아래는 GDPR식 LIA를 복붙하지 않고 국내 개정 PIPA 실제 구조로 재작성한 초안이며, 조문은 법무 검증 전제다.

### 2.1 전제: 수집 대상의 법적 성격
다크웹/유출사이트 게시 데이터는 "자유롭게 처리 가능한 공개정보"가 아니라 **타인의 위법행위로 유통되는 개인정보**일 수 있다. "공개성"이 처리근거가 되지 않으며, "보기만 한다"도 면책이 아니다(매칭·저장·표시 모두 PIPA상 처리). 마스킹된 `account_masked`도 결합 시 식별 가능하면 개인정보다.

### 2.2 처리근거 — 활동별로 **분리** (정당한 이익에 뭉뚱그리지 않음)

| 처리 활동 | 처리근거(법무 확정 전 초안) | 비고 |
|---|---|---|
| 자사 4개 도메인 키워드 매칭·노출 탐지 | **개정 PIPA 정당한 이익 조항(§15①(7) 초안)** + 전자금융감독규정 보안의무 | 국내 정당한 이익은 GDPR식 일반조항이 **아니며** "정보주체 권리보다 명백히 우선 + 합리적 관련성 + 예측가능성" 3요건을 좁게 요구 → 법무가 요건 충족을 축조 입증 |
| 임직원 계정 노출 처리 | **근로관계·취업규칙·정보보호서약 기반 사전 고지** (정당한 이익에 섞지 않음) | 모니터링 사실을 취업규칙/서약에 반영해 **예측가능성·고지** 확보(노무·법무 검토) |
| 고객 식별·금융거래정보 노출 처리 | **신용정보법 §19/§20 안전조치 의무** + PIPA | 신용정보회사 등에 해당 → 신용정보법 **우선·가중 적용**, 목적 외 이용 절대 금지 |

> **국내 정당한 이익 3요건 균형기록(연 1회 + 소스 변경 시 재작성·재승인)**: ① 목적 정당성(금융소비자 보호·법령상 보안의무), ② 수단 최소성(자사 한정 매칭·벌크 미수집·평문 미저장·제3자 즉시 레닥션), ③ 권리 균형(자사 정보주체 보호 이익이 명백히 우선, 제3자 PII는 "수집하지 않음"으로 균형). **GDPR LIA가 아니라 국내 요건으로 작성한다.**

### 2.3 신용정보법 우선적용 — 보고 시한·수신처까지 구체화
원칙 선언에 그치지 않고 대응 플로우에 시한·수신처를 적시한다(법무가 사건유형별 의무발생 매트릭스 사전 확정):

| 노출 유형 | 발동 의무(초안) | 수신처·시한 |
|---|---|---|
| 고객 개인정보 유출(자사 책임 유출 시) | PIPA 유출통지·신고 | 개인정보보호위/KISA, 요건·시한 법무 확정 |
| 고객 **신용정보** 누출 | 신용정보법 보고 | 금융위/금감원 — 별도 라인·시한 법무 확정 |
| 전자금융 침해사고 징후 | 전자금융거래법·전자금융감독규정 보고 | 금융보안원/금감원 |

> **"PIPA만 가정한 통지 설계"는 과소대응이다.** 신용정보가 섞이면 PIPA 유출통지뿐 아니라 금융위/금감원/금융보안원 보고가 **별도 발동**된다. 본 모니터링의 산출(노출 발견)이 곧 통지·신고를 의미하진 않으나, **법무가 사건마다 의무발생 여부를 판단**하도록 핸드오프 포인트를 둔다.

### 2.4 수집 가능/불가의 명확한 선

**GREEN (허용)**: 4개 도메인/사명/브랜드 매칭 리스팅 항목만 — `account_masked`, `domain`, `breach_name/title`, `breach_date`, **`data_classes`는 분류 라벨만(실값 아님)**, `severity`, `source`, `reference_url`(포인터).

**RED (금지)**: 전체 덤프 다운로드·보관 / 제3자 PII 벌크 / 평문 비밀번호·전체 카드·주민·계좌번호 원문 / 인증·CAPTCHA 게이트 뒤 접근·우회 / 마켓 거래·덤프 구매·샘플 요청 / 타인 계정 로그인·자격증명 유효성 검증.

**YELLOW (사전 법무 판단)**: 협력업체/제휴사 계정 매칭 / 자사 이메일로 가입한 외부 SaaS 유출(자산 노출 vs 임직원 사적 가입) / 신규 다크웹 소스 추가(소스별 적법성 사전 심사).

### 2.5 처리위탁·국외이전·외부 인텔 벤더 — **해시 프리픽스 강제**
- 외부 인텔 벤더·Supabase는 **수탁자**일 수 있음 → 위탁계약 필수(목적·범위 한정, 재위탁 제한, 안전조치, 목적 외 이용금지, 감사권, 종료 시 파기), 위탁사실 처리방침 공개.
- **외부 조회 시 자사 임직원/고객 이메일 원문 송신은 설계상 금지(권고가 아니라 금지).** HIBP/XposedOrNot 등은 **해시 프리픽스(k-anonymity range) 조회만** 허용. 원문 송신/해외 리전 불가피 벤더는 위탁계약 + 국외이전 고지/공개 + **데이터 출처 실사(범죄마켓 구매분 소비 차단)**를 계약 게이트로 강제.
- **Supabase 리전은 국내로 고정**해 국외이전 트리거를 최소화한다.

### 2.6 증거능력 vs 최소보존 — 위법수집 충돌 해소가 선행
- **이원화 저장**: 운영 DB(마스킹·분류, 일반 보존 후 파기) / 증거 보존고(legal hold — 법무 명시 지시 건만, 해시·타임스탬프·provenance, 접근 법무 한정).
- ⚠️ **자기모순 해소 게이트**: chain-of-custody 증거보존고는 **수집 자체의 적법성(2.7 §48 의견서)이 확인된 후에만** 설계·운영한다. 적법성 미확정 상태의 증거 체계화는 위법수집증거 배제 리스크 + "위법수집을 인지·체계화한 정황"을 남길 수 있다.
- **그래도 넘지 않는 선**: 증거 목적이라도 평문 비밀번호 원문·제3자 PII 벌크·구매 덤프는 보존하지 않는다.

### 2.7 ⚠️ 형사 적법성 — Phase 1 진입의 필수 산출물
.onion DLS '공개 리스팅 읽기'가 **정보통신망법 §48(접근권한 없는 정보통신망 침입)**에 해당하는지에 대한 **명시적 법무 의견서**를 Phase 1 진입 게이트의 필수 산출물로 격상한다.
- 랜섬웨어 갱단 .onion DLS는 "누구나 보라고 게시한 공개 웹"과 법적 성격이 다를 수 있고, **robots/공개 여부가 접근권한을 부여하지 않는다.**
- 함께 검토: 통신비밀보호법, 해외 사이트 형사관할·국제예양.
- "공개 리스팅이니 읽기는 합법"이라는 전제를 **법무가 문서로 확인하기 전에는 단일 파일럿도 시작 금지.**

### 2.8 승인 체계(RACI) · 정기 재검토
- **RACI**: 도입 승인(CISO+법무/DPO+정보보호위원회) / 소스·벤더 변경(보안팀장 신청→법무 사전심사→CISO 승인) / 노출 대응(보안팀+침해 시 CPO/신용정보관리·보호인) / 외부 벤더 계약(법무+정보보호+구매).
- **재검토**: 거버넌스 리뷰 연 1회 · 소스/벤더·법령 변경 시 즉시 · 침해 대량발견 사건단위 · 접속기록 점검 분기. 결과 문서화·서명 보존.

## 3. 아키텍처

전용 Tor 수집 계층은 **기존 darkweb-monitor-dashboard에 손대지 않는다**(기존은 합법 API만, Edge/Actions에서 동작 — Tor 라우팅 없음). 물리·논리 분리된 전용 수집 계층이 결과만 Supabase REST로 단방향 push 한다.

```
┌─────────────────────────────────────────────────────────────────────┐
│  격리 수집 VPC (회사망과 피어링 없음 · 별도 클라우드 계정 · 전용 egress)  │
│                                                                       │
│  점프호스트(bastion, MFA/SSH/세션로깅) ─── 운영자 유일 진입로            │
│                                                                       │
│  scheduler ─▶ job queue ─▶ crawler workers (rootless, cap-drop=ALL)   │
│                                  │ socks5h://tor-gw:9050              │
│                          Tor 게이트웨이 컨테이너 (client only, exit 아님)│
│                                  │  ↓ Tor (.onion 만)                  │
│  normalizer/redactor ◀─ raw staging(tmpfs,TTL)  (스트리밍 라인매칭;     │
│  4도메인 매칭·마스킹·leak-guard      비매칭 라인 즉시 폐기, 덤프 미저장)    │
└───────────┬───────────────────────────────────────────────────────────┘
            │ HTTPS / Supabase REST (적재 전용 최소권한 키, 단방향 push)
            │ egress allowlist = {Tor 디렉터리/가드, Supabase 도메인} 만
            ▼
   Supabase (국내 리전) — breach_findings / scan_runs
            │  ↑ source_tier='darkweb' 행만 stale-delete (전역 DELETE 금지)
            │  ↑ DB CHECK: source_tier='darkweb' AND account<>'' INSERT 거부
            │  ↑ DB 제약: domain ∈ 4개 화이트리스트 · RLS 역할기반 + MFA
            ▼
   기존 Next.js 대시보드 (Phase 1 필수 프런트 변경: account_masked 강제 ·
                          reference_url 비활성 배지 · source_tier 배지/필터)
```

- **격리·네트워크**: 별도 클라우드 계정(blast radius 격리), 회사망 피어링 없음, egress allowlist 3종만. clear-net 직접 접속 차단. `.onion` 해석은 `socks5h`로 Tor에 위임(DNS 누수 방지), redirect를 clearnet으로 따라가지 않음.
- **Tor 계층**: tor 데몬 전용 컨테이너, `IsolateSOCKSAuth` 출처별 회로 격리. NEWNYM 회전은 정상 실패 백오프 수준만(**공격적 회전 금지**). **Tor relay/exit 미운영(클라이언트로만).**
- **크롤러**: Python `httpx`(async, socks5h) 기본. 헤드리스(Playwright)는 예외적으로만, **CAPTCHA/로그인 게이트 감지 시 즉시 중단(우회 안 함)**.
- **스케줄·큐·복구**: 출처당 동시성 1, 지수 백오프+jitter, 연속 실패 cooldown, DLQ, 출처별 try/catch 격리, `finding_id` 멱등 upsert.

## 5. 데이터 최소화 · PII · 저장 스키마 매핑

### 5.1 키워드-온리 매칭
- `domain_exact`(4개), `email_pattern`(`@(jbfg\.com|jbbank\.co\.kr|kjbank\.com|wooricap\.com)`), `subdomain`(ib/ebanking/vpn…), `brand_kw`(전북은행/광주은행/JB금융/JB우리캐피탈/JBFG/KJB), `staff_domain_pattern`(**명부 평문 미반출 — 솔트+HMAC-SHA256 해시셋만**).
- `brand_kw` 단독은 finding을 만들지 않는다 — 같은 윈도우에 도메인/이메일/자격증명 토큰 **동반(co-occurrence)** 시에만 승격(일반 게시물 벌크 흡입 방지).

### 5.2 스트리밍 라인 매칭 — 전체 덤프 미저장 보장
청크→라인→링버퍼 `ctx[i-N…i+N]`(N=2)만 RAM 유지. 비매칭 라인 즉시 폐기(디스크 미접촉). **서킷 브레이커**: 매칭률 비정상 시 소스 격리+사람검토 큐. ⚠️ **필수변경**: 서킷브레이커·co-occurrence의 임계값·테스트셋·false-negative 검증셋을 Phase 1 산출물로 명세(미검증 시 사실상 벌크수집).

### 5.3~5.4 컨텍스트-온리 추출 · 제3자 PII 레닥션
- 윈도우 자체도 영속 저장 안 함(분류·`account_masked` 생성에만 쓰고 폐기). 분류기는 **값 미저장**, 유형만 한글 라벨.
- 자사 매칭 계정→부분 마스킹 / 제3자 이메일→**기본 저장 안 함** / 평문 비번→**폐기**(라벨만) / 비번 해시→유형만 / RRN·카드→즉시 레닥션(값·마스킹값 미저장).
- ⚠️ **주체 오판 방어**: 주체 식별은 도메인/이메일 정규식 + 임직원 HMAC 양성확인으로만, 불확실 매칭은 승격 안 함(검토 큐). ⚠️ **HMAC 솔트 위험**: 기본 미저장, 불가피 시 솔트 분리·회전 + 법무 가명/익명 판단.

### 5.5 처리 순서 불변식 (코드 강제)
```
매칭 → 윈도우 → [PII 탐지 → 제3자 레닥션 → 비번/RRN/카드 폐기] → 분류 카탈로그
  → 정규화(고정 키셋) → leak-guard 어서션 → REST 적재 → 버퍼 zeroize
```

### 5.6 스키마 매핑 — ⚠️ 키셋 정합화(PGRST102 자기위반 해소)
> **검증된 사실**: 실제 `mkRawFinding`는 12개 키만 반환하고 normalizer 의사코드가 가정한 일부 키를 emit하지 않음 → 벌크 upsert PGRST102. **`mkRawFinding`과 darkweb normalizer 양쪽을 동시에** 수정해 키 집합을 강제 동일화하기 전 다크웹 적재 금지.

| breach_findings 컬럼 | 다크웹 매핑 |
|---|---|
| `finding_id` | `sha1(domain\|alias\|listing\|sourceId)` (기존 규칙 동형) |
| `account_masked` | `we***@jbbank.co.kr` 또는 도메인 집계 시 `*@domain` |
| `account` | **`""`(빈값 강제)** — 다크웹 식별 평문 미저장 |
| `domain` | 4개 자사 도메인 — **DB 제약 화이트리스트 강제** |
| `data_classes` | 한글 라벨 배열(실값 없음) |
| `source` / `reference_url` | `다크웹 (사이트라벨)` / 출처 포인터(쿼리스트링 자격증명 절단) |

### 5.7 leak-guard — 앱 + **DB 제약** 이중화
앱: 적재 직전 정규식 재스캔(비번/RRN/카드 잔존 시 거부+알람). ⚠️ **필수변경**: **DB CHECK/트리거로 `source_tier='darkweb' AND account<>''` INSERT 거부** + `domain` 화이트리스트 제약(앱+DB 이중 방어).

## 6. 기존 대시보드 통합 — ⚠️ "무변경" 주장 철회

### 6.1 평문 노출 경로 차단 (필수)
- **(a)** `DashboardClient.tsx:94`의 `accountMasked: r.account || r.account_masked`는 평문 `account`를 우선 렌더 → 다크웹 행은 `account` 무시·`account_masked` 강제.
- **(b)** DB 레벨 `source_tier='darkweb' AND account<>''` INSERT 거부(적재 원천 차단).
- **(c)** `reference_url` 렌더는 무조건 클릭가능 `<a>` → .onion/http 스킴 검사해 **비활성 배지("Tor 필요")**로(분석가 단말 직결 클릭 차단).

### 6.2 stale-delete tier 격리 — Phase 1 "첫 코드 변경"
> **검증된 충돌**: 현재 `sbDeleteStale`는 tier 필터 없는 **전역 DELETE** → 다크웹 행이 다음 surface 스캔에 전멸(역도 성립).
> **원자적 3종 동시 배포**: ① `007_source_tier.sql`(`source_tier TEXT NOT NULL DEFAULT 'surface'`+인덱스), ② `sbDeleteStale`에 `source_tier=eq.surface` 필터, ③ surface upsert에 `source_tier:'surface'` 주입. **하나라도 누락 시 상호 전멸 → 이 변경 전 다크웹 적재 절대 금지.**

### 6.3 출처 배지·필터 (관리자 전용)
`select`에 `source_tier` 추가, 다크웹 행 별도 배지(보라/검정), 상단 `전체/표면웹/다크웹` 세그먼트 필터, 계열사별 위험 tier별 분리. RLS는 역할분리 보강 전제(7장).

| **P11** | Tor relay/exit 운영 | **클라이언트로만** |

| **P13** | 외부 인텔 API에 자사 이메일 **원문 송신** | **해시 프리픽스(k-anonymity) 조회만**, 국내 리전 |
| **P14** | 노출 데이터를 자사 방어 외 목적 이용·재배포 | 자사 방어 한정 |
| **P15** | 법무·CISO·DPO 승인 없이 다음 Phase 진입 | 진입 게이트 준수 |
| **P16** | 다크웹 파이프라인이 surface/infostealer 행을 전역 DELETE | `source_tier='darkweb'` 행만 stale-delete |

> **P7 판정**: rate-limit 준수·단일 동시연결·UA 위조 금지·접근거부 존중 = 허용. CAPTCHA/JS챌린지/안티봇을 *뚫는* 모든 기법 = 금지. **모호하면 금지로 간주, 법무 질의.**

- **Kill-switch 3계층**: L1 소스 토글 · L2 수집기 정지 · L3 인프라 격리(egress 차단+키 rotate). 자동: 게이트 감지→L1 / PII 폭증→L2 / 영장·중단요청→L3+증거보전.
- **감사(불변 로그, 현재 미존재→도입 필수)**: `collection_audit`(소스·매칭키워드·추출/폐기 건수, append-only) · `access_audit`. 직무분리(수집≠감사).
- **역할분리**: Service Owner(보안팀장, kill-switch) / Operator(Tor 호스트·적재키) / Analyst(읽기) / Auditor(읽기전용) / Legal·DPO(사인오프, 시스템 권한 없음).

## 8. 단계적 로드맵 · 진입조건

### Phase 0 — 클리어넷 공개소스 (현 상태 + 거버넌스·접근통제 보강)
- 범위: 소스 C(API들) + 소스 A(ransomware.live류). Tor 불필요. 공수 2~3주·0.3 FTE.
- **마무리 조건(현재 미충족)**: ① **RLS 역할분리**(현재 전원 동일권한), ② **MFA**(현재 비번 단독), ③ **감사테이블·도메인 화이트리스트 DB 제약**(현재 미존재), ④ 거버넌스 CISO 승인·금지목록 PR 체크리스트화, ⑤ **외부 API 해시 프리픽스 전환**+Supabase 국내 리전.

### Phase 1 — Tor `.onion` DLS 읽기 (분리 전용 인프라, 소스 B)
- 범위: 랜섬웨어 DLS .onion 공개 리스팅 읽기전용(자사 키워드만). 공수 6~10주·0.5~1 FTE+법무.
- **진입 게이트 — 4종 전부 충족 전 단일 파일럿도 금지**: (a) **형사 적법성 법무 의견서**(§48 등), (b) **평문/링크 노출 차단 코드**(6.1), (c) **stale-delete tier 격리**(6.2 원자 3종), (d) **보존 컬럼·키셋 정합화**(5.6+아래). + CISO·DPO 승인, 드라이런 P1~P16 위반 0.

### Phase 2 — 공개 포럼/채널 인덱스 (가장 신중, 소스 F·D·E)
- 인증·초대제 전면 제외(P1). 진입: 소스별 Legal+DPO **개별** 사인오프, 분기 감사 2회 무위반, **빌드 vs 바이 재평가**.

### ⚠️ 보존정책 빌드가능화 (Phase 1 (d))
> **검증된 사실**: 실DB(001~006)에 `last_seen`/`first_seen` 컬럼이 **없다**(`discovered_at`·`updated_at`만) → 설계 §5의 보존 pg_cron 잡 빌드 불가.
> **택1**: (A) `last_seen`/`first_seen` 추가 마이그레이션, 또는 (B) 보존 기준을 `discovered_at`/`updated_at`으로 재작성. pg_cron 삭제 잡과 Supabase 자동백업 보존창 일치(물리삭제/crypto-shred). **보존 미구현 상태 운영 진입 금지.**

보존(초안, 법무 확정): 다크웹 finding 180일 / `scan_runs` 365일 / 윈도우·원문 0일(미저장) / evidence vault 법무 지정(crypto-shred).

## 9. 빌드 vs 바이 권고

| 항목 | 자체구축 | 상용 Buy | **하이브리드(권장)** |
|---|---|---|---|
| 법적 리스크 | **높음**(수집책임 JB) | 낮음(벤더 책임) | 낮음(자체는 공개소스만) |
| 커버리지 | 좁음(공개 리스팅) | 넓음(폐쇄포럼·마켓) | 넓음 |
| 운영 인력(연) | 0.5~1 FTE | 0.2 FTE | 0.3~0.5 FTE |

- **폐쇄 마켓·인증 포럼은 본 설계의 절대금지(P1·P4) 때문에 자체구축으로 합법적으로 못 본다** → **상용(Recorded Future/Flashpoint/KELA/SpyCloud) 또는 금융보안원·금융ISAC 피드** 구독으로 보완이 정도.
- **자체구축의 진짜 가치 = 비용통제된 공개소스 상시 모니터링 + 자사 맞춤 트리아지·대시보드(이미 구축) + 데이터 주권.** → **하이브리드**: 공개 클리어넷/Tor 리스팅은 자체, 폐쇄영역은 구독, 결과는 기존 대시보드 통합. 자체 Tor의 ROI는 "DLS 조기경보 한 점"으로 좁힌다.

## 10. 법무·경영진 결정 필요 오픈이슈

1. ⚠️ **[법무·최우선] 정당한 이익 조문·요건 재확정** — 국내 PIPA엔 GDPR식 일반 정당한 이익 조항 부재. §15①(7) 및 요건을 법무가 축조 확인.
2. ⚠️ **[법무·Phase 1 게이트] .onion 직접 읽기 형사 적법성 의견서** — 정보통신망법 §48·통신비밀보호법·관할. "공개=접근권한"은 미검증 전제. 미발급 시 파일럿 금지.
3. **[법무] 위법수집증거 충돌** — 적법성 미확정 상태 evidence vault 운영 보류 여부.
4. **[법무·DPO] 사건유형별 의무발생 매트릭스** — PIPA/신용정보법/전자금융 보고 시한·수신처 사전 정의.
5. **[법무·노무] 임직원 모니터링 사전고지** — 취업규칙·서약 기반 분리.
6. **[법무] 외부 SaaS 유출 구분 기준**.
7. **[법무·구매] 위탁·국외이전·벤더 실사** — 이메일 원문 송신 전면 금지(해시 프리픽스만) 확정, 벤더 데이터 출처 실사.
8. **[법무·DPO] 제3자 PII 임시처리·HMAC 다이제스트** — PIA 필요 여부, 가명/익명 판단.
9. **[법무] evidence vault·보존** — legal hold 권한·기간·crypto-shred, 운영 DB 일반 보존기간.
10. ⚠️ **[경영진·CISO] 접근통제 투자 승인** — 역할기반 RLS+MFA+감사테이블+도메인 화이트리스트 DB 제약을 Phase 0 마무리 조건으로.
11. **[관리] 자산 소유·운영 귀속** — Public 레포·개인 GitHub·개인 도메인(jbax.co.kr) 운영 중인 현 자산을 JB 공식 자산으로 이관할지, Tor 인프라 소유·service_role 키 책임 소재.
12. **[관리] 출처 카탈로그 큐레이션** — 모니터링 onion/클리어넷 승인 주체·변경관리, `reference_url`에 .onion 저장·표시 여부.
13. **[CISO] 운영 게이트** — 본 인프라는 설계 단계. 실제 운영 개시 전 별도 운영 승인·법무 사인오프를 게이트로.
